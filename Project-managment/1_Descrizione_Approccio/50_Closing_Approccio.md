# 50 — Closing: Approccio (Ziradò)

Scopo: definire criteri di accettazione e processo di chiusura con audit e report finale.

## 1) Criteri di Accettazione e Collaudi
- Evidenze: test plan eseguito, beta feedback, analytics base attivi, checklist store soddisfatte.
- Pass/Fail: crash rate < 1%, performance P95 < 2s, contenuti ≥ soglie, approvazione docente.
- Pendenze: note e piano di risoluzione post‑chiusura (se presenti).

## 2) Consegna Formale e Handover
- Consegna codice, doc tecnica/utente, credenziali servizi, piano manutenzione minimo.

## 3) Chiusura Amministrativa
- Verifica costi vivi (hosting, domini), chiusura ambienti test non necessari.

## 4) Audit Post‑Implementazione
- Confronto KPI vs baseline (adozione, rating, contenuti), lezioni apprese, raccomandazioni.

### 4.1 Struttura delle lezioni apprese
- Processo: cosa ha funzionato (es. timebox, review), cosa migliorare (stima capacità, lead time store).
- Prodotto: funzionalità più usate, attrito UX, performance.
- Persone: collaborazione, comunicazione, carico, crescita competenze.

## 5) Knowledge Transfer e Archiviazione
- Archiviazione repo/tag release finale, backup asset, doc finale in cartella progetto.

## 6) Rapporto Finale di Progetto
- Executive summary, risultati, performance, allegati principali (POS, PDS, WBS, Gantt, Budget, Accettazione Cliente).

### 6.1 Narrazione di chiusura
Il rapporto finale racconta brevemente la storia del progetto (problema iniziale, ipotesi, rilasci, risultati), evidenziando come l’approccio scelto abbia ridotto il rischio e massimizzato l’apprendimento. Serve anche come base per le fasi di scaling.

## 7) Checklist
- [x] Accettazione firmata
- [x] Handover completato
- [x] Audit e lezioni apprese
- [x] Rapporto finale emesso
